# Deployment runbook

Deploys WatchTodo to **$0/month** hosting: Django backend (Docker) + React static
frontend on **Render**, Redis on **Upstash**. WebSockets (`wss://`) work on Render's
free web service. See [`specs/08-deployment.md`](specs/08-deployment.md).

```
React (Render static/CDN) ──REST+wss──▶ Django (Render web, Docker) ──▶ Redis (Upstash)
T-Watch Ultra ────────────REST+wss──────────────────▲
```

## 1. Redis (Upstash, free)

1. Create a database at <https://console.upstash.com> (Redis, any region near the
   backend).
2. Copy the connection URL. Use the **`rediss://…` TLS URL** (Upstash requires TLS).
   This is your `REDIS_URL`.

> Upstash free tier persists data and is enough for one user. Both the note data
> and the Channels layer share this one instance.

## 2. Backend + frontend (Render Blueprint)

1. Push this repo to GitHub (done).
2. Render → **New + → Blueprint** → select the repo. Render reads
   [`render.yaml`](render.yaml) and creates two services:
   `watchtodo-backend` (Docker web) and `watchtodo-web` (static site).
3. Fill the env vars marked "to set" (they are intentionally not in git):

   **watchtodo-backend**
   | Var | Value |
   |-----|-------|
   | `REDIS_URL` | the Upstash `rediss://…` URL |
   | `API_KEY` | a long random string (e.g. `openssl rand -hex 24`) |
   | `CORS_ALLOWED_ORIGINS` | the frontend URL, e.g. `https://watchtodo-web.onrender.com` |
   | `DJANGO_SECRET_KEY` | auto-generated by Render |
   | `DEBUG` | `false` (preset) |

   `ALLOWED_HOSTS` is filled automatically from `RENDER_EXTERNAL_HOSTNAME`
   (handled in `settings.py`) — no need to set it.

   **watchtodo-web** (build-time, baked into the bundle)
   | Var | Value |
   |-----|-------|
   | `VITE_API_BASE` | `https://watchtodo-backend.onrender.com` |
   | `VITE_WS_BASE` | `wss://watchtodo-backend.onrender.com` |
   | `VITE_API_KEY` | same value as backend `API_KEY` |

4. Deploy. First the backend, then set the web service's `VITE_*` to the backend URL
   and **trigger a redeploy of the frontend** (Vite bakes these at build time, so
   they must be set before the build that ships).

### Verify
```bash
curl -s https://watchtodo-backend.onrender.com/api/health        # {"status":"ok"}
KEY=<API_KEY>
curl -s -X POST https://watchtodo-backend.onrender.com/api/notes \
     -H "X-API-Key: $KEY" -H 'Content-Type: application/json' -d '{"text":"hello prod"}'
# Open the web URL in two tabs -> live updates. Then point the watch at the URLs.
```

## 2b. Frontend on Vercel (alternative to Render static)

Vercel hosts the React frontend nicely; it **cannot** host the backend (the Django
app is a persistent ASGI server with WebSockets — not serverless). Deploy the
backend on Render/Railway/Fly first, then:

1. Vercel → **Add New → Project** → import this repo.
2. **Root Directory: `frontend`** (Vercel reads `frontend/vercel.json`: Vite preset,
   `npm ci` / `npm run build` → `dist`, SPA rewrite).
3. Environment Variables (Production), then deploy:
   | Var | Value |
   |-----|-------|
   | `VITE_API_BASE` | `https://watchtodo-backend.onrender.com` |
   | `VITE_WS_BASE` | `wss://watchtodo-backend.onrender.com` |
   | `VITE_API_KEY` | same as backend `API_KEY` |
4. Set the backend's `CORS_ALLOWED_ORIGINS` to the Vercel URL
   (e.g. `https://watchtodo.vercel.app`) and redeploy the backend.

If you use Vercel for the frontend, you can delete the `watchtodo-web` service from
`render.yaml` (keep only `watchtodo-backend`).

> Vercel CLI path: `cd frontend && npx vercel` (login once), set the env vars with
> `npx vercel env add VITE_API_BASE` etc., then `npx vercel --prod`.

## 3. Cold-start keepalive

Free web services sleep when idle; the first request then waits for a wake-up
(seconds). To keep it warm:

- Set repo **variable** `HEALTH_URL` =
  `https://watchtodo-backend.onrender.com/api/health` (Settings → Secrets and
  variables → Actions → **Variables**). The included
  [`.github/workflows/keepalive.yml`](.github/workflows/keepalive.yml) pings it
  every ~12 min.
- Or use a free external pinger (UptimeRobot, cron-job.org) on the same URL.

## 4. Point the watch at prod

Provision on-device (MENU → Wi-Fi setup) with:
- API base `https://watchtodo-backend.onrender.com`
- WS base `wss://watchtodo-backend.onrender.com`
- API key = backend `API_KEY`

(Or pre-seed `firmware/include/secrets.h` and reflash.) For production, prefer
pinning the host root CA in the firmware (`TLS_INSECURE 0`) over `setInsecure()`.

## Alternatives

- **Frontend on Netlify / Cloudflare Pages / Vercel** instead of Render static:
  build `frontend`, publish `dist/`, set the same `VITE_*`. Update the backend
  `CORS_ALLOWED_ORIGINS` to that origin.
- **Backend on Railway / Fly.io / Koyeb**: deploy `backend/Dockerfile`, set the same
  env, ensure the platform passes WebSocket `Upgrade` headers (all of these do).

## Troubleshooting

- **WS won't connect in prod**: confirm `wss://` (not `ws://`) and that
  `CORS_ALLOWED_ORIGINS` includes the exact frontend origin. The proxy `Upgrade`
  path is exercised locally in `docker-compose.proxy.yml` (see below).
- **400 Bad Request / DisallowedHost**: the host isn't trusted — on Render this is
  auto-handled; elsewhere set `ALLOWED_HOSTS`.
- **First request slow**: cold start — see §3.

### Local proxy parity check (optional)
`docker-compose.proxy.yml` puts an nginx reverse proxy in front of the backend with
the same `Upgrade`/`Connection` headers a host uses, so you can confirm the WS path
before deploying:
```bash
docker compose -f docker-compose.yml -f docker-compose.proxy.yml up --build
# WebSocket via the proxy: ws://localhost:8080/ws/notes/?key=<API_KEY>
```
